=========================================================
PROJECT: CAMPUS PULSE - ENHANCED TECHNICAL EXPLANATION
=========================================================

1. PROJECT OVERVIEW
-------------------
Campus Pulse is an automated, paperless Event Management and Engagement Ecosystem designed for modern universities. It solves the problem of manual registration, tedious attendance tracking, and delayed certification. The system integrates Biometric Face ID, QR-based verification, and real-time data analytics into a single, cohesive web platform.

2. THE "SMART CAMPUS" CONCEPT
-----------------------------
What makes Campus Pulse "Smart" is the elimination of manual human intervention in the event lifecycle:
- Login: No passwords needed (Face ID).
- Registration: Instant confirmation and QR delivery.
- Attendance: One-second QR scan replaces manual sign-in sheets.
- Certification: Instant PDF generation and email delivery upon attendance.

3. CORE TECHNICAL WORKFLOW (THE PIPELINE)
-----------------------------------------
A. FACIAL BIOMETRIC ENROLLMENT:
   When a user registers, the system doesn't just store a photo. It uses the "face-api.js" library (powered by TensorFlow.js) to detect 68 facial landmarks. These landmarks are converted into a "Face Descriptor"—a 128-float numerical array (a digital fingerprint of the face).

B. IDENTITY VERIFICATION (THE LOGIN):
   During Face ID login, the system captures a live frame from the webcam, generates a new descriptor, and sends it to the Node.js backend. The server performs a "Euclidean Distance" calculation against the stored descriptors in MongoDB. If the distance is below a specific threshold (e.g., 0.6), the user is authenticated.

C. EVENT LIFECYCLE & CAPACITY LOGIC:
   Faculty can create events with a defined "Capacity." The system implements a safety check: before allowing a student to register, it queries the database to ensure (Registered Count < Max Capacity). This prevents over-crowding and ensures resource availability.

D. DYNAMIC QR GENERATION:
   Upon registration, a unique Registration ID is generated in MongoDB. This ID is encoded into a QR code using the 'qrcode' library. This acts as a "Digital Ticket" that is both displayed on the student dashboard and sent to their registered email.

E. AUTOMATED ATTENDANCE & PDF ENGINE:
   When the Faculty scans the student's QR code, the system triggers two simultaneous actions:
   - It updates the 'attended' status in the database to 'true'.
   - It invokes the "PDFKit" engine on the server to dynamically generate a Participation Certificate. The student’s name, event title, and date are "drawn" onto a PDF template in memory, which is then instantly emailed via Nodemailer.

4. THE TECH STACK (WHY WE CHOSE IT)
-----------------------------------
- NODE.JS & EXPRESS: Chosen for its non-blocking I/O, which is essential for handling multiple simultaneous registrations and certificate generations.
- MONGODB ATLAS: A NoSQL database that allows us to store the 128-bit Face Descriptor arrays natively without performance overhead.
- FACE-API.JS: Enables high-speed facial recognition directly in the browser using the GPU, reducing the load on the server.
- GSAP & LENIS: Used to provide a "Cinematic" user experience with smooth scrolling and high-end animations, making the platform feel modern and premium.

5. SECURITY & DATA INTEGRITY
----------------------------
- PII Protection: We do not store raw images of users; we only store mathematical descriptors. This ensures that even in the event of a data leak, a user's actual face cannot be reconstructed.
- Role-Based Access Control (RBAC): The system strictly separates Student and Faculty dashboards. A student cannot access the attendance scanner or event creation tools.

6. API ENDPOINT DOCUMENTATION (THE BACKBENE)
------------------------------------------
The system communicates through a RESTful API architecture. Here is the breakdown:

A. AUTHENTICATION & IDENTITY:
- POST /register: Handles faculty sign-up, including the storage of the 128-bit Face Descriptor.
- POST /login-face: Receives a live Face Descriptor and performs Euclidean Distance matching for biometric entry.
- POST /login: Standard email/password fallback authentication.
- POST /create-user: Utility endpoint to programmatically create student accounts.

B. EVENT MANAGEMENT:
- GET /events: Dynamically fetches all events and calculates their status (Upcoming/Ongoing/Completed) based on the server time.
- POST /create-event: Allows faculty to define event parameters (Capacity, Venue, Department).
- PUT /update-event/:eventId: Enables real-time modification of event details.
- DELETE /delete-event/:eventId: Performs a cascaded delete, removing the event and all associated registrations.

C. REGISTRATION & ATTENDANCE:
- POST /register-event: Links a student to an event, generates a unique QR code, and triggers the confirmation email.
- POST /mark-attendance: The most critical endpoint. It verifies the QR code, updates the database, generates the Participation PDF, and emails it to the student.
- GET /registrations/:studentId: Retrieves a student's history of event participation and certificates.

D. ANALYTICS & REPORTING:
- GET /attendance-analytics: Compares "Registered" vs "Attended" counts for every event.
- GET /participation-trend: Aggregates registration data into monthly buckets for growth tracking.
- GET /department-activity: Breaks down campus participation by department (CS, IT, Mech, etc.).
- GET /dashboard-stats: Provides high-level numbers (Total Users, Events, Registrations) for the admin overview.

7. CONCLUSION
-------------
Campus Pulse isn't just a website; it’s an automation engine. By combining AI (Face ID) with IoT-like functionality (QR Scanning) and Cloud Automation (Instant Emailing), it transforms how universities manage their student engagement data.
